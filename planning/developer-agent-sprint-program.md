# AgentFarm Developer Agent Sprint Program

## Purpose
Convert the MVP Developer Agent roadmap into sprint-wise execution documents that teams can run in sequence without breaking MVP guardrails.

## Scope Guardrails
1. One Developer Agent role for MVP delivery track.
2. Approved connectors only: Jira, Microsoft Teams, GitHub, company email.
3. Approval-first governance for medium and high risk actions.
4. Full action and evidence traceability for every executed task.

## Program Assumptions
1. Sprint length is two weeks.
2. No platform-wide refactor required to start.
3. Planning-first and evidence-first delivery remains mandatory.

## Sprint Sequence
1. Sprint 1: Trust And Execution Core.
2. Sprint 2: Adoption And Reliability Scale.
3. Sprint 3: Memory And Compliance Packaging.

## Sprint Documents
1. planning/sprints/sprint-1-trust-and-execution-core.md
2. planning/sprints/sprint-2-adoption-and-reliability-scale.md
3. planning/sprints/sprint-3-memory-and-compliance-packaging.md

## Entry Gate Per Sprint
1. Previous sprint critical defects are triaged and assigned.
2. Story-level KPI instrumentation plan is defined.
3. Security and policy impact review is complete.
4. Rollback plan is defined for every mutation story.

## Exit Gate Per Sprint
1. Story acceptance criteria pass in staging.
2. Required audit fields are complete.
3. Security and policy checks pass.
4. KPI instrumentation is live before closure.
5. Pilot feedback is captured for every story.

## Prioritization Summary
1. Highest priority: Sprint 1 stories S1-1, S1-2, S1-3, S1-4.
2. Next priority: Sprint 2 stories S2-1, S2-2, S2-4.
3. Then expand: Sprint 3 stories S3-1, S3-2, S3-3, S3-4.

## Delivery Rule
Execute sprints in order. Do not start Sprint 2 implementation until Sprint 1 exit gate is passed; do not start Sprint 3 implementation until Sprint 2 exit gate is passed.
